# Findings 模板（可选）

当当前工作存在并行 review、测试、排查、反馈或临时观察时，创建 `doc/findings.md`，并按“范围 -> 结论 -> Findings -> 说明”的结构记录。

```md
# Review

## 1. 范围

- 评审目标：这次要对照什么、检查什么。
- 覆盖范围：本次实际检查了哪些目录、模块、页面、链路或文档。
- 对照基线：本次拿什么作为基线；不适用写 `N/A`。
- 评审方式：静态对照 / 编译验证 / 运行验证 / 其他。
- 排除项：本次明确不检查什么；不适用写 `N/A`。

## 2. 结论

- 总结论：先给一句总体判断。
- 当前有效偏差：写清还有几个有效偏差；若无，写“本次未发现有效偏差”。
- 摘要：按高到低概括剩余偏差。

## 3. Findings

### 3.1 [P2] 示例标题

- 现状：当前代码、行为或结构是什么。
- 基线：期望对齐的对象或口径是什么。
- 证据：列出关键文件、关键事实或关键现象。
- 影响：说明为什么这是偏差，以及影响什么。
- 建议：并入当前 task / 新建 task / 暂存 / 废弃。

### 3.2 [P3] 示例标题

- 现状：...
- 基线：...
- 证据：...
- 影响：...
- 建议：...

## 4. 说明

- 限制：本次没做什么，例如未编译、未联调、未跑回归。
- 备注：其他需要保留的说明；不适用写 `N/A`。
```

## 使用要求

- `doc/findings.md` 是一份并行评审记录，不是 task，也不是提交单元。
- 先写“范围”和“结论”，再列 Findings，避免只堆零散问题点。
- Findings 按严重度排序；若没有有效偏差，`## 3. Findings` 下明确写“本次无有效 Findings”。
- 每条 finding 只保留 5 类信息：现状、基线、证据、影响、建议。不要扩展成过程流水账。
- 证据优先写文件路径、接口、行为差异、日志摘要或静态对照事实，不写空泛判断。
- 建议只保留 4 种去向：并入当前 task、新建 task、暂存、废弃。
- 若某条 finding 最终沉淀为长期有效的约束、已知问题或稳定口径，再按需要回写 `doc/project.md`。
