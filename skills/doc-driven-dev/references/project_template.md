# 项目主文档

> 这是整个项目的主文档，不是某一次 task 的执行记录。
> 第 1 章只写短、硬、稳定的全局规则。
> 其余章节用于说明项目介绍、目的、路线和设计；具体实施进入 task 后再写 `tasks.md` 与 `task-*.md`。

## 1. 全局规则

- 交互与编码：例如语言、编码、格式与命名约定。
- 工作方式：例如文档先行、`task` 是最小实施单元、`finding` 为可选并行记录。
- 修改边界：例如只读源、可修改区、禁止触碰区。
- 回写规则：例如哪些变化回写 `project.md`、哪些变化回写 `task` 或 `findings`。
- 提交规则：例如提交描述来源、标识要求与阻断条件。
- 文档指针：只列需要优先看的核心文档，例如 `doc/tasks.md`、`doc/task-编号-概述.md`、`doc/findings.md`。

## 2. 项目介绍

- 项目是什么：一句话说明项目定位。
- 背景：当前为什么做这件事，以及已确认的上下文事实。
- 现状：当前项目已经具备什么，还存在哪些关键问题。
- 范围：这个项目主要覆盖哪些对象、模块、流程或交付物。

## 3. 项目目的

- 为什么做：这件事的核心动机或收益。
- 总目标：项目长期希望达到什么状态。
- 当前阶段目标：当前阶段要先解决什么问题。
- 非目标：明确当前不做什么，避免 scope 漂移。

## 4. 路线图

- 阶段划分：项目整体准备分几步推进。
- 当前阶段：当前所处阶段，以及这一阶段的重点。
- 里程碑：关键里程碑、优先级或推进顺序。
- 进入 task 的条件：哪些事项在进入具体实施后需要拆到 `tasks.md` 与 `task-*.md`。

## 5. 设计

- 总体设计：项目整体采用什么思路或结构推进。
- 当前阶段设计：当前阶段已经确定的设计方案。
- 长期有效结论：会长期保留的规则、边界、术语、接口、约束或已固定资产。
- 专题文档索引：如有独立专题设计文档，在这里维护链接或索引。

## 6. 附加信息（按需保留）

- 权威来源：需求、现有代码、上游文档、用户说明、外部规范等。
- 已知限制：历史包袱、暂不处理项、长期保留风险、兼容性约束或已知问题。
- 术语表：需要统一口径的关键术语。
- 相关索引：其它需要长期保留的项目级索引。
