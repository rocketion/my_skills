# Handoff

## 1. 定位

这是一个上下文交接 skill，用于在当前对话过长、需要切换新会话、需要交给另一个 agent、或需要把当前工作状态固化下来时，生成一份 `handoff.md`。

`handoff.md` 不是普通聊天摘要，而是下一个对话或下一个 agent 的“重启点”。

目标是让接手方无需用户重复背景，就能基于文档直接继续当前工作。

## 2. 适用场景

当用户表达以下意图时，应使用本 skill：

- handoff
- 交接
- 生成 handoff.md
- 压缩上下文
- 整理上下文给下个会话
- 我要开新对话继续
- 导出当前上下文
- 把当前工作状态保存下来
- 把当前进度给下一个 agent

## 3. 不适用场景

以下情况不应使用本 skill：

- 用户只需要普通摘要。
- 用户只需要一段简短说明。
- 当前任务已经完全结束，且不需要继续执行。
- 交接对象不需要技术细节、执行状态或下一步动作。

## 4. 核心理念

本 skill 借鉴 Claude Code context compact 的设计思想：

```text
模型总结式压缩 + 工作状态重建
```

也就是：

1. 先把长对话中的用户意图、技术细节、决策、错误、反馈和当前状态压缩成结构化文档。
2. 再把继续工作必须依赖的状态重新写入文档。
3. 最后给出明确的恢复指令，使下一个 agent 能直接从中断点继续。

因此，`handoff.md` 的目的不是“读起来好看”，而是“接手后能继续干活”。

## 5. 核心原则

1. 保留用户真实意图，而不是只记录字面请求。
2. 保留最近工作状态，最近上下文优先级高于早期闲聊。
3. 保留技术细节，尤其是路径、命令、错误、配置、函数、类名、接口、约束和结论。
4. 保留用户反馈，尤其是用户纠正、否定、抱怨和明确偏好。
5. 保留未完成任务和下一步动作。
6. 对旧内容可压缩，但不得压掉会影响继续工作的事实。
7. 不编造未知内容；不确定时明确写“未知”或“未确认”。
8. 不写无关铺垫，不把 handoff 写成泛泛总结。

## 6. 执行前分析

在写入 `handoff.md` 之前，必须按时间顺序梳理当前对话，识别以下内容：

1. 用户的显式请求与真实意图。
2. 当前任务的目标、边界和验收口径。
3. 已经采取的处理路径和关键动作。
4. 已讨论或使用的技术概念、工具、框架、协议和 API。
5. 已查看、修改、创建或计划修改的文件、目录、函数、类、配置和命令。
6. 已经做出的设计决策，以及被拒绝的方案。
7. 遇到的错误、失败尝试和修复方式。
8. 用户的纠正、偏好、约束、风格要求和不满点。
9. 当前完成了什么、还缺什么、卡在哪里。
10. 下一个 agent 应该立即执行的单一最佳下一步。

## 7. 产物位置

默认产物为当前工作目录下的：

```text
handoff.md
```

如果用户指定其他文件名或路径，则使用用户指定的文件名或路径。

如果当前项目已有文档体系，应优先遵守项目约定；但交接文档仍必须具备本 skill 要求的继续工作信息。

## 8. handoff.md 标准结构

生成的 `handoff.md` 必须使用以下结构。

```md
# Handoff

## 1. 主要请求与真实意图

说明用户的显式请求。

进一步说明用户真正想完成什么，而不是只复述字面问题。

记录关键约束、偏好、背景和用户已经明确否定过的方向。

## 2. 当前目标

说明当前阶段正在推进的目标。

本节要让接手方知道应该优化什么，以及不要偏离到哪里。

## 3. 关键技术上下文

列出当前工作涉及的重要技术概念、工具、框架、协议、API、模型、仓库或系统。

每个条目说明它为什么与当前工作有关。

## 4. 文件与代码位置

列出所有重要文件、目录、函数、类、配置、命令或代码段。

每个条目至少包含：

- 路径或名称
- 作用
- 当前已知状态
- 重要细节或片段

如果没有实际文件，也要写明“当前尚未确认具体文件”。

## 5. 已确定的决策

记录已经做出的决定。

包括：

- 采用的方案
- 不采用的方案
- 不采用的原因
- 后续不应重复争论的结论

## 6. 错误、修复与用户反馈

记录：

- 遇到的错误
- 失败的尝试
- 如何修复或为什么失败
- 用户纠正过的点
- 用户不满意的回答方向
- 用户明确要求的表达方式或工作方式

本节的目的，是防止下一个 agent 重复犯错。

## 7. 当前工作状态

精确描述交接前最后正在做什么。

必须包含：

- 最近一条用户请求
- 最近一次 assistant 已执行或准备执行的动作
- 已完成事项
- 未完成事项
- 当前阻塞点
- 仍需读取、修改或验证的内容

## 8. 待办任务

用可执行的条目列出剩余任务。

每一项都应具体到接手方可以直接开始。

## 9. 推荐下一步

写出下一个 agent 应该立即执行的单一最佳下一步。

该步骤必须直接对应用户最近的请求。

如果有必要，引用用户最近的原话。

不要在这里启动与当前请求无关的旧任务。

## 10. 给下一个 agent 的继续指令

写给接手方的操作指令。

说明如何基于本 handoff 继续，不要让用户重复上下文。

必须包含类似以下含义：

- 直接从本 handoff 继续。
- 不要要求用户重新解释前文。
- 除非用户要求，不要泛泛复述 handoff。
- 从推荐下一步开始执行。

---

## Resume Directive

继续从中断点推进当前工作。

不要要求用户重复上一段对话。

不要泛泛总结本 handoff，除非用户明确要求。

以上文为事实来源，从“推荐下一步”开始执行。
```

## 9. 压缩规则

压缩旧上下文时遵守以下规则：

1. 旧闲聊可以高度压缩。
2. 旧但仍影响当前工作的技术结论必须保留。
3. 最近工作必须高保真保留。
4. 用户最近的请求和反馈必须优先保留。
5. 文件路径、命令输出、错误信息、配置项、接口名和函数名不得随意改写。
6. 对已完成但仍影响后续理解的工作，写结论，不写流水账。
7. 对未完成工作，写清剩余动作和阻塞点。
8. 对不确定内容，明确标记“未知”或“未确认”。

## 10. 状态重建规则

`handoff.md` 必须重建接手方继续工作所需的状态。

至少应覆盖：

- 当前目标
- 当前工作目录或仓库
- 涉及文件
- 已知约束
- 用户偏好
- 已做决策
- 已失败方案
- 剩余任务
- 推荐下一步

如果当前任务涉及代码，还应尽量覆盖：

- 相关文件路径
- 相关函数、类、模块或配置
- 已运行命令
- 已见错误信息
- 测试或验证状态
- 未验证风险

## 11. 写作风格

`handoff.md` 应该：

- 使用中文。
- 简洁但不丢关键细节。
- 技术事实优先，少写客套话。
- 对复杂内容使用列表。
- 对路径、命令、代码、配置使用代码块。
- 明确区分事实、推断和未知。
- 不写无用铺垫。

## 12. 禁止事项

禁止：

1. 把 handoff 写成泛泛的聊天摘要。
2. 省略当前状态和下一步。
3. 让下一个 agent 重新问用户背景。
4. 编造没有出现过的文件、命令或结论。
5. 把用户已经否定的方案重新包装成建议。
6. 用英文写主要内容，除非用户特别要求。

## 13. 最小完成标准

一次合格的 handoff 至少满足：

1. 接手方能知道用户到底要什么。
2. 接手方能知道当前做到哪里。
3. 接手方能知道哪些坑不能再踩。
4. 接手方能知道哪些文件、命令或系统最重要。
5. 接手方能直接执行下一步，而不是重新询问用户。

## 14. 推荐生成提示词

当需要生成 handoff 时，可使用以下提示词作为执行口径：

```text
你的任务是创建一份详细的 handoff.md，用于让新的对话或新的 agent 继续当前工作。

这份 handoff 需要充分保留技术细节、代码模式、架构决策、错误、修复方式、用户反馈和当前工作状态。它的目标不是普通总结，而是让接手方无需用户重复上下文，就能继续开发或继续分析。

在写最终文档之前，请按时间顺序分析当前对话，识别：

1. 用户的显式请求与真实意图。
2. 已经采取的处理方式。
3. 关键决策、技术概念和代码模式。
4. 文件名、函数签名、命令、配置和代码片段等具体细节。
5. 遇到的错误以及修复方式。
6. 用户反馈，尤其是纠正、否定和“不要这样做”的要求。
7. 用户明确要求继续推进的待办事项。
8. 交接前正在做的精确工作。
9. 与用户最近请求直接一致的下一步。

将结果写入 handoff.md。

要求：

- 使用中文。
- 不要写泛泛摘要。
- 保留实现相关细节。
- 最近上下文比早期上下文保留得更完整。
- 必要时引用用户最近原话。
- 不编造未知事实。
- 不确定内容标记为“未知”或“未确认”。
- 让下一个 agent 可以直接继续，不需要用户重新解释。
```

## 15. 完成后的响应

生成或更新 `handoff.md` 后，向用户简要说明：

1. 已生成或已更新的文件路径。
2. handoff 覆盖了哪些核心内容。
3. 推荐下一个对话如何使用该文件。

不要在最终回复中重复整份 `handoff.md`，除非用户明确要求。
